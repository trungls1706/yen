📘 REACT NATIVE SENIOR – FULL INTERVIEW PREP (Phần 1-3)1. React & React Native – Nền tảngCâu hỏiTrả lời chuẩn SeniorGhi chú quan trọngReact Native khác React JS?React JS render ra DOM (Browser). React Native render ra Native UI Components (ví dụ: UIKit trên iOS / Android View trên Android).React Native không có HTML/CSS thật; dùng StyleSheet cho styling và Yoga cho layout (Flexbox).React Native có Virtual DOM không?Có, nhưng nó dùng React Reconciler để tạo ra Shadow Tree (bản nháp của native view tree), sau đó map sang native views.Cốt lõi của cơ chế reconciliation (so sánh và cập nhật) là giống React web, nhưng output là native module.Giao tiếp (Communication) trong RN?Performance phụ thuộc vào cơ chế giao tiếp JS Thread ↔ Native Thread (gọi là Bridge hoặc cơ chế mới JSI - JavaScript Interface).JSI (được sử dụng trong New Architecture/Fabric) cho phép giao tiếp đồng bộ và hiệu quả hơn Bridge.2. Performance – Câu hỏi rất hay gặp ⚡❓ Nguyên nhân gây lag trong React Native?Re-render không cần thiết: Component render lại dù props hoặc state không thay đổi.Anonymous function trong render: Tạo ra hàm mới (() => {}) sau mỗi lần render, làm phá vỡ cơ chế React.memo của component con.FlatList không optimize: Thiếu keyExtractor, getItemLayout, hoặc windowSize sai.Xử lý nặng trên JS Thread: Bất kỳ tính toán phức tạp nào (parsing JSON lớn, loop nặng) làm block JS Thread và gây giật lag cho UI (đặc biệt là animation).Image quá lớn/không tối ưu: Tải ảnh có độ phân giải cao hơn mức cần thiết.Too many listeners / timers: Gây rò rỉ bộ nhớ (memory leak) và lãng phí tài nguyên CPU.❓ Tối ưu FlatList như thế nào?Kỹ thuậtMục đíchkeyExtractorGiúp React định danh và theo dõi các mục, tối ưu hóa quá trình reconciliation.getItemLayoutCung cấp chiều cao/rộng của mỗi item, cho phép FlatList tính toán vị trí cuộn mà không cần render tất cả item (tăng hiệu suất cuộn).initialNumToRenderGiảm số lượng item render ban đầu để hiển thị nhanh hơn.windowSizeĐiều chỉnh số lượng item được giữ trong bộ nhớ (đã render) bên ngoài viewport.removeClippedSubviews(Chỉ Android) Cải thiện hiệu suất bằng cách không vẽ các item ngoài màn hình.Tránh Inline Component:Không tạo component ngay trong prop renderItem (renderItem={() => <MyItem />}).❓ useCallback / useMemo dùng khi nào? (Senior answer)Nguyên tắc: Không dùng bừa. Lạm dụng dẫn đến chi phí bộ nhớ (memory overhead) và giảm khả năng đọc code.Dùng khi:useCallback: Khi bạn truyền hàm callback xuống component con đã được bọc bởi React.memo. Điều này giúp ngăn component con render lại vô ích.useMemo: Khi bạn cần tính toán một giá trị nặng (ví dụ: lọc mảng lớn, tính toán phức tạp) và giá trị đó chỉ nên được tính lại khi dependencies thay đổi.❓ React.memo có luôn tốt không?❌ Không.Drawback: Nó chỉ thực hiện shallow comparison (so sánh nông) trên props. Nếu props là object/array mới được tạo ra trong mỗi lần render của component cha, React.memo sẽ thất bại và component con vẫn render, trong khi còn phải chịu thêm chi phí so sánh props.3. State ManagementCâu hỏiPhân tích Senior MindsetLựa chọn Công cụKhi nào dùng local state, khi nào Redux?Áp dụng tư duy “Right tool for right job”.UI state (isOpen, isToggle): useState. Complex state (Form phức tạp): useReducer. Cross-screen/Global state (Theme, Auth Status): Redux / Zustand. Built-in thunk, dễ dàng tạo slice.Redux vs React Query?Redux quản lý Client State (logic app). React Query quản lý Server State (data fetching, caching, revalidation, sync).Không thay thế nhau; thường dùng cả hai hoặc dùng React Query để loại bỏ nhu cầu quản lý Server State trong Redux.
